如果你把大模型（LLM）想象成一个只认识数字、不认识文字的“数学怪兽”，那么 **Tokenization（分词/分标记）** 就是那个连接人类文明与数学怪兽的“万能翻译官”。

### 1.1 为什么大模型不直接读文字？
计算机的本质是二进制运算。大模型在底层处理的是复杂的向量（Vector）和矩阵（Matrix）。对于模型来说，汉字“猫”和英文“Cat”在最初只是两个无法理解的图像或符号。

**Tokenization 的核心使命：** 将不连续的、形状各异的文本，切碎成模型可理解的最小语义单元（Token），并给每一个单元贴上一个唯一的“身份证号”（ID）。

---

### 1.2 分词技术的进化史：从“词”到“碎块”

为了找到最完美的“切分点”，科学家们经历了三个阶段：

#### 1.2.1 词级别分词 (Word-level)
*   **做法：** 按照空格和标点切分。
*   **例子：** `"I love apples"` -> `["I", "love", "apples"]`
*   **缺点：**
    *   **词表爆炸：** 英语有几十万个词，加上各种变体（apple, apples, apple's），模型需要背诵一个无穷大的字典。
    *   **未知词（OOV）：** 如果出现一个字典里没有的新词（如：ChatGPT），模型会直接“宕机”，显示为 `<UNK>`。

#### 1.2.2 字符级别分词 (Character-level)
*   **做法：** 把每个字母或汉字拆开。
*   **例子：** `"Cat"` -> `["C", "a", "t"]`
*   **缺点：**
    *   **语义丧失：** 单个字母 "C" 没有任何意义。
    *   **序列过长：** 一篇千字文章拆成字母可能有几万个，模型处理起来极慢且容易“忘掉”开头。

#### 1.2.3 子词级别分词 (Subword-level) —— **当下的黄金标准**
*   **核心逻辑：** 常用词完整保留，不常用词拆分成有意义的“零件”。
*   **例子：** `"unhappiness"` -> `["un", "happi", "ness"]`
*   **优点：** 完美平衡了词表大小和语义完整性。即便遇到新词，也可以通过“零件”拼凑出来。

---

### 1.3 核心算法：BPE (Byte Pair Encoding) 深度拆解

目前最主流的模型（如 GPT、Llama）大多采用 BPE 算法。它的原理非常像“消消乐”：**不断合并出现频率最高的字符对。**

#### 【通俗演练】BPE 是如何炼成的？
假设我们的语料库里只有三个词：`"hug"`, `"pug"`, `"pun"`, `"bun"`。

1.  **初始状态：** 拆成字母：`h u g`, `p u g`, `p u n`, `b u n`
2.  **统计频率：** 发现 `u` 和 `n` 经常在一起出现。
3.  **合并：** 将 `u`+`n` 合并为一个新 Token `un`。
4.  **循环：** 发现 `p` 和 `un` 经常在一起，合并为 `pun`。

**伪代码逻辑展示：**
```python
# BPE 训练简化逻辑
def train_bpe(corpus, num_merges):
    # 1. 初始词表：所有基础字母
    vocab = set(all_characters_in(corpus))

    for i in range(num_merges):
        # 2. 统计语料库中相邻两个 Token 的频率
        pairs = get_stats(corpus)
        # 3. 找到频率最高的组合 (比如 'u' 和 'n')
        best_pair = max(pairs, key=pairs.get)
        # 4. 合并这个组合，并加入词表
        corpus = merge_pair(best_pair, corpus)
        vocab.add(best_pair)

    return vocab
```

---

### 1.4 词表（Vocabulary）到底是什么？

**词表（Vocab）** 就是大模型的“新华字典”。它是一个映射表，记录了每个 Token 对应的唯一 ID。

| ID | Token | 类型/备注 |
| :--- | :--- | :--- |
| 0 | `<PAD>` | 占位符（用于对齐长度） |
| 1 | `<BOS>` | 句子开始 |
| 50256 | `GPT` | 常用完整词 |
| 50257 | `un` | 语义子词（前缀） |
| 50258 | `ness` | 语义子词（后缀） |

**词表的作用：**
1.  **索引：** 把人类文字转换成数字索引（ID）。
2.  **映射：** 模型最后预测出一个 ID，再根据词表查出对应的文字还给人类。

---

### 1.5 Tokenization 的全流程输入与输出

当你输入一句话给模型时，后台发生的过程如下：

#### Step 1: 预分词 (Pre-tokenization)
*   **输入：** `"Hello, world!"`
*   **输出：** `["Hello", ",", "world", "!"]`（按空格和标点粗拆）

#### Step 2: 子词切分 (Subword Splitting)
*   **输入：** `["Hello", ",", "world", "!"]`
*   **逻辑：** 查 BPE 合并规则，看是否有更小的碎块。
*   **输出：** `["Hello", ",", " world", "!"]`（注意：空格通常会被转化为一个特殊符号 `_` 或保留在 Token 开头）

#### Step 3: ID 映射 (Encoding)
*   **输入：** `["Hello", ",", " world", "!"]`
*   **逻辑：** 在词表中查找对应的 ID。
*   **输出：** `[15496, 11, 995, 0]`（这一串数字就是喂给 Transformer 的“食物”）

---

### 1.6 工业界的进阶：字节级 BPE (BBPE)
早期的 BPE 遇到中文或德文会很头疼，因为字符集太大了。
现在的模型（如 Llama 3）使用 **Byte-level BPE**：
*   它不再以字符为单位，而是直接处理 **UTF-8 编码的字节（Bytes）**。
*   **意义：** 任何语言、甚至损坏的文本或图片，都可以被拆解为 256 个基础字节的组合。这意味着模型词表可以非常精简，且永远不会出现未知词（No more `<UNK>`!）。

---

### 1.7 进阶视角：Token 的工程学与经济学

在实际工业应用中，Tokenization 不仅仅是分词，它直接决定了模型的**运行成本**和**多语言能力**。

#### 1.7.1 语言的“肥沃度” (Token Fertility)
**为什么同样的文字，中文 API 费率往往感觉比英文贵？**
这就是“Token 肥沃度”的概念：即一个单词/汉字平均消耗多少个 Token。
*   **早期模型 (如 Llama 2)**：词表只有 32k，且偏向英文。一个中文字可能被拆成 2-3 个字节 Token。
*   **现代模型 (如 Llama 3/GPT-4o)**：词表扩展到 128k 甚至更高。通过优化中文语料比重，一个中文字通常只需 0.6~0.8 个 Token。
*   **直观影响：** 词表越大，模型对非英语言的压缩率越高，意味着同样的 Context Window（上下文窗口）能塞进更多内容，推理速度也更快。

#### 1.7.2 工业实现：Tiktoken vs SentencePiece
*   **SentencePiece (Google/Meta)**：将空格视为普通字符（如 `_`）。它不依赖于语言特定的分词器，非常适合多语言。
*   **Tiktoken (OpenAI)**：基于 BPE 的极致优化实现。它引入了复杂的正则表达式来预处理文本，特别是在处理代码缩进（连续空格）时，它能将其合并为单个 Token，极大地节省了代码生成任务的消耗。

#### 1.7.3 临界点：从数字 ID 到几何向量 (The Bridge)
当你拿到一串 ID（如 `[15496, 11]`）时，Tokenization 的使命就完成了。
接下来，这些 ID 会进入 **Embedding Layer（嵌入层）**。
*   **核心逻辑：** 每一个 ID 对应一个高维向量（例如 4096 维）。
*   **本质：** 这就像在数学空间里给每个 Token 找一个坐标。语义相近的词（如“猫”和“喵”），它们的坐标在空间中也会靠得更近。

### 1.9 深度问答：词表的生命周期与工业标杆

在实际开发中，词表的设计是一个“一损俱损”的过程。以下是两个最核心的工程疑问：

#### 1.9.1 词表是在什么时候生成的？
词表的生成发生在 **Step 0：模型训练前的基建阶段**。
1.  **语料采样**：从海量数据中抽取代表性样本（如几百 GB）。
2.  **分词器训练**：在样本上运行 BPE/BBPE 算法，生成词表和合并规则。
3.  **终身锁定**：一旦词表生成，它就与模型的输入/输出层（Embedding 层）权重深度绑定。
**关键推论**：一旦模型开始预训练，词表就不能再改动。如果强行修改（如增加 10 个 Token），模型必须从零开始重新训练，否则所有已学到的知识都会发生偏移。

#### 1.9.2 业界主流模型词表对比
目前不同模型在“词表大小”上呈现出明显的派系差异：

| 模型家族 | 词表大小 | 核心算法 | 工业评价 |
| :--- | :--- | :--- | :--- |
| **Llama 2** | 32,000 | SentencePiece (BPE) | **轻量派**：对英文友好，但中文等语言压缩率低。 |
| **GPT-4** | 100,256 | Tiktoken (BBPE) | **黄金标杆**：对代码和多语言做了极佳平衡。 |
| **Llama 3** | 128,256 | Tiktoken (BBPE) | **现代派**：兼顾了多语言效率与显存占用。 |
| **Gemma** | 256,000 | SentencePiece (BPE) | **激进派**：追求极高压缩率，但浪费了大量模型参数存储词表。 |

**结论：** 并非越大越好。词表越大，模型输出层的计算开销（Softmax）越大，且会侵占宝贵的模型权重参数空间。**10w~15w** 是目前大模型在智能、速度与资源占用之间的“最优解”。

#### 关键总结：“跨模态融合与极限压缩”的新战局

1. **从“单字对抗”到“词组垄断”：** 早期模型争夺的是“把一个汉字用多少个 Token 拼凑”。而在如 **Qwen 3.6** 和 **Kimi 2** 的分词器中，高频中文字词、成语甚至短句已直接编入独立 Token。一段话曾经需要 20 个 Token，现在已被压缩至少数“超级 Token”。
2. **多模态与世界语的大一统：** **GPT-5** 与 **Gemini 3.1** 级别的分词系统，内部映射规则已经不再单纯区分输入的是中文、英文还是图像波段与声讯。依靠极大扩容的基础词表，它们将人类各种信息载体对齐到了同一高维空间，彻底消除了语言间的“API 过路费”鸿沟。
3. **“算力护城河”向下沉淀：** 虽然维护高达甚至超过 30万级别的庞大词表，会导致模型输出层（Softmax）遭遇海量计算负担，但最新的稀疏计算和路由结构平摊了这种灾难。厂商的设计共识演变为：**通过超级词表极致提高上下文密度，再靠先进集群架构硬扛算力开销。**

### 1.11 深度剖析：GPT-5 与 Gemini 3.1 的底层分词器与“巨型词表”的代价

针对最新一代巨兽模型（如 GPT-5 和 Gemini 3.1），它们的分词器在基础算法层面是否相对独立？是否仍在沿用传统的 BBPE？如此庞大（30万~50万+）的词表又会对模型物理层产生什么深远影响？

#### 1. 算法流派：殊途同归的“大一统”，但并不交叉
虽然外界常统称为多模态切词器，但二者的底层架构血脉截然不同，且系统高度独立封闭：

*   **GPT-5 (跨模态 Tiktoken 系统)**：
    *   **底层渊源**：GPT 系列从骨子里依然是 **BBPE (Byte-Level Byte Pair Encoding)** 的史诗级魔改版。OpenAI 的 Tiktoken 本质上是一个用极度优化的底层语言编写的正则+BBPE 匹配引擎。
    *   **独立性变革**：在 GPT-5 的世代，它不再仅仅对 UTF-8 的纯文本字节做合并。它首创地将视频帧（Image Patches）、声学高频振幅（Audio Spectrograms）乃至机器操作指令统统视为同源的序列流数据（Sequence of bits）。GPT-5 的“新 BBPE”不仅在寻找高频的字母对字组，甚至在进行**“这帧表情”加上“这句叹词”的跨模态联合高频打包**。
*   **Gemini 3.1 (连续模态流 Tokenizer)**：
    *   **底层渊源**：Google 从始至终是 **SentencePiece 体系 (以 Unigram 和 BPE 混合算法为主)** 的宗师。Gemini 沿用且大幅度强化了这套逻辑。
    *   **独立性变革**：SentencePiece 的核心理念是**把空格和标点看作普通字符**，不依赖任何特定语言的预处理（如切割英文单词）。到了 Gemini 3.1，Google 开辟了“无需离散切分”的端到端范式。它就像一个精通全语言的速记员，可以直接生吞各种乱码文本与音频流，并基于深度的统计概率切分出对应的离散向量集合。

**结论**：它们在多模态上都实现了大一统，但在核心工程基座上依旧南辕北辙。一个（Tiktoken）信仰极其暴力的显式合并规则约束，一个（SentencePiece）信仰概率最大化的端到端黑盒。

#### 2. 超庞大词表 (30万~50万级别) 的“双刃剑”效应

传统词表一般在 **6万 (Claude 早期)** 到 **10万 (GPT-4 早期)** 之间游走。当我们将词表暴力拉高到 30万甚至突破 50万规模时，这不仅是分词能力的进步，更是底层硬件和模型架构的生死局：

**显著的红利（巨头为何拼命扩大词表）：**
1.  **物理极限下的“空间折叠”**：传统大模型一个中文成语要耗费至少 3-4 个 Token。在大词表时代，“不知不觉”、“流体力学”等常用复合词组直接化身单个超级 Token。你的 200 万超长窗口，在有效信息容量上可能抵得上过去几千万。
2.  **万国平权与跨模态基座**：海量词表终于装得下世界上的小众语言、极其生僻的数学公式、以及天文数字级别的像素编码块。没有任何语言和数据再需要遭受被“掰碎解析”的痛苦，全人类语料获取了系统级的平权。

**恐怖的物理代价（小厂为什么不敢做 >20万 的词表）：**
1.  **输出点名层（Softmax）的算力灾难**：生成大模型的本质是在最后一层做一个全词表的“多项选择题”。如果你的词汇库有 50万个词条，模型在输出**每一个 Token 时**，都必须计算这 50 万个选项的概率分布。这涉及极其庞大的矩阵乘法 `[Hidden_Dimension × 500,000]`，会直接榨干推理用的显存（VRAM），并成倍增加文字的吐出延迟（TTFT）。
2.  **黑洞般的参数侵占**：大字典非常占地方。每个独立 Token 都要在输入 Embedding 层和输出 Softmax 层注册独立的向量。以目前千亿模型的隐藏层维度（如 8192 维）来算，**挂载一个 50万规模的词表，仅处理字典本身就需要占用惊人的约 16 GB 显存容量（对应整整约 80多亿参数！）**。对于小模型而言，你的参数全拿去存字典了。
3.  **罕见分支的“冷冻失效”**：词表过大，意味着不可避免地收录了大量低频词汇。如果训练数据不够恐怖，很多极度偏门的 Token 在长达半年之久的预训练中可能只会被更新寥寥几次，导致它们的理解能力停留在弱智水平，严重污染模型的精密回路。

**最后防线：** 到了 2026 年世代，头部厂商能够无视这些算力开销和显存黑洞，背后仰仗的是极其成熟的 **稀疏分布式路由网络（MoE）** 和万卡级别的恐怖算力池。他们把算力倾斜到架构搭建上，换取了文本容量的跨维度飞升。
